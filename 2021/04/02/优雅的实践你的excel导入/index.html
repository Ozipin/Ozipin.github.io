优雅的实践你的excel导入

发表于 2021-04-02 | 分类于 java
将excel导入全过程实现的层次分明，简单明了，体验友好

导言

​ excel文档的批量导入在我们的实际开发过程中经常会遇到。由于批导的功能一般较为复杂，除了繁复的导入数据处理外，还有各种数据格式的验证，业务数据的验证，乃至于错误信息的收集等等无一不在加深代码的复杂度。所以批导的功能很多人会写，但却只有很少的人能将自己的批量导入的功能写的层次分明，实现优雅。本文将介绍自己目前对于excel文件导入功能的处理的全过程。

模板下载

​ excel的导入必须要让使用者知道批量导入的时候可以填写哪些内容，需要填写哪些内容，具体的内容怎么填。一个说明准确，提示清晰的模板会减少大量的用户使用疑问。下文会介绍两种模板的下载实现方式和各自实现方式的优缺点

​ 两种实现方式都能实现导入模板的下载，但是由于实现方式的不同，两种方式之间也会存在着各自的长处和缺点。

​ 直接生成方式：由于模板文件是在每次下载的时候通过代码生成的，所以对于开发者而言，模板的生成只需要编写简单的easy excel的导出modal类即可。对于提醒用户的必填，非必填之类的也可以通过easy excel的注解如表头名ExcelProperty在原有表头名字段后面备注必填或非必填来提醒用户知道哪些字段需要填写；或者通过表头背景颜色即铜鼓注解HeadStyle的fillForegroundColor属性来进行提示，比如定义红色为必填，黄色为选填，则可以设置fillForegroundColor的值为10（红色），或者13（黄色）来达到提供用户的目的。此外此modal类一般也会用于导入的功能，所以也会降低一些导入功能改动而引起问题的概率。这样看来，用直接生成的方式似乎很美好，为什么还要有通过静态资源直接下载的方式？相比于静态资源的直接下载，直接生成的下载方式薄弱的地方在于excel的生成，即我们无法通过代码生成太过复杂的excel文档或者用来生成太过复杂的文档对于当前的excel生成方案来说太过复杂。举个例子：我们要让用户导入学生的数据，学生的性别可能是男或者女，一般而言让用户手动输入是完全没问题的，但是为了更好的用户体验，我们需要把这个输入改成一个下拉框，下拉框的值是男/女，当我们使用直接生成的方式的时候可能就意味着我们需要针对这个下拉框来做额外的编码工作，甚至于当所选方案生成下拉框实现较为困难时会花费我们更多的时间。

​ 静态资源下载：由于这种方式的下载实质就是下载一个我们提前准备好的模板文件，所以我们在编码工作之外还需要去动用excel来新建一个模板文档，这自然不如使用直接生成的方式来的简单。此外，由于excel文档是静态资源而非实时生成的，这样它和导入modal类之间的关联性就没有直接生成方式的那么强，这样在有些需求变动的时候，两边出现不一致的情况的几率可能就相对较大。但是使用excel软件来生成模板文件的话，确实也是给了我们更大的便利，简单如限制excel输入框的格式，复杂的如一个用代码实现起来比较困难的下拉框你可以在这边简单的实现，甚至于级联下拉框，更甚者可以在excel模板里面加入vba脚本。

​ 两种方式都可实现一个导入模板的下载，我们在实际开发中可以根据实际情况选择实现起来比较简单的方式。此外，在遇到两种方式实现起来都较为困难的情况时，我们甚至于可以将两种方式进行结合，比如我这边曾经遇到需要将一个下拉框的数据是根据数据库实时获取的，所以我先用静态资源维护了一个excel模板，下拉框填充的内容指向excel的另一个sheet的指定列，然后在模板下载的时候，实时写入最新的数据到另一个sheet的指定列。

​ 完成了基本的数据格式验证之后一般而言这些通过验证的数据已经是可靠的数据了，但是很多时候，正确输入的数据并不一定是业务逻辑上的合法数据，所以很多时候我们还会有业务逻辑上的验证。以学生信息为例，正常来说一个系统里的一个学号的学生应该是只有一个的，如果只信任导入的数据不做业务逻辑的验证，那么最后导入的结果就会有问题。所以除了基础的数据格式验证之外，我们还需要有一些业务逻辑的验证，并收集相关的错误信息。此处暂时没有比较好的选择，所以目前都是手写的代码。举例代码如下：

​ 理论上来说，我们的数据验证和准备录入的数据的收集已经做得差不多了，将数据进行落库就完成了整个导入的过程。但是由于数据量的不同，可能落库的时机会有所不同，对于数据量较小的我们可以在doAfterAllAnalysed方法里面进行保存，即数据完全验证完毕后一次性全部入库。对于数据量太大的情况就需要考虑在invoke方法中选择对验证通过的数据列表分批落库了，比如当validModals里面每有一千条数据的时候就进行一次保存。

​ 此外，由于easy excel针对数据的处理是按行来进行的，所以我们每次的数据业务逻辑验证都是针对的数据库已有的数据来的，即两条完全一样的数据按照上面的代码的验证逻辑来说的话，都会通过验证。所以我们在实际验证的时候也要考虑这种情况。解决这种问题的方式有两种： 第一种就是加快数据落库的频率，没验证通过一条数据就落库一次，这样可能会稍稍加大导入时数据库的压力；第二种就是缓存此次导入的所有数据的特征值，比如不允许学号重复，就缓存此次导入的所有学号，用于对后续的数据的验证。

​ 至此，我们的数据导入的大体功能已经完成了，数据可以通过校验，我们能收集到数据校验不通过的原因，能对校验通过的数据进行落库保存。

返回结果

​ 一般接口而言，我们对于处理完的请求，返回相关的数据json，由前端进行结果提示就好。但是对于导入而言，这样的方式可能会有稍微的不妥，因为直接简述成功了多少行失败了多少行，对于用户进行改正然后再导入的过程是很不便的，不知道哪些行实际成功了哪些行实际失败了，就无从改正以进行后续的录入；而如果将所有的错误信息进行返回，则数据量稍微一大，整个的提示信息就很很长，用户阅读起来也会显得麻烦，更重要的是对于用户重新整理自己的导入文件也很不便。
